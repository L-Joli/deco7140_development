Accessibility Assessment

This assessment outlines how can the client's website meet accessibility standards based on WCAG guidelines. Each principle includes success criteria and the actions taken to ensure inclusivity for all users.

Principle: Perceivable

  • 1.1 Text Alternatives
    • 1.1.1 Alternative Text for Non-text Content

      All non-text content that is presented to the user has a text alternative that serves the equivalent purpose, except for the situations listed below.

      • Actions: Community stories and cultural images will include descriptive alt text and aria-label so that users with visual impairments can fully engage with the content.
  • 1.3 Adaptable
    • 1.3.2 Meaningful Sequence

      When the sequence in which content is presented affects its meaning, a correct reading sequence can be programmatically determined.

      • Actions: The site uses semantic HTML to ensure a logical reading order.
  • 1.4 Distinguishable
    • 1.4.1 Avoid Color-only Cues

      Color is not used as the only visual means of conveying information, indicating an action, prompting a response, or distinguishing a visual element.

      • Actions: Add icons or text labels along with colour.
    • 1.4.3 Contrast (Minimum)

      The visual presentation of text and images of text has a contrast ratio of at least 4.5:1, except Large Text, Incidental and Logotypes.

      • Actions: High-contrast colours are applied to all text and buttons to ensure resources and events are perceivable for older users and those with vision impairments.
    • 1.4.4 Resize Text

      Except for captions and images of text, text can be resized without assistive technology up to 200 percent without loss of content or functionality.

      • Actions: The site uses responsive design so text and layouts adjust seamlessly when resized, supporting accessibility on mobile and desktop.

Principle: Operable

  • 2.1 Keyboard Accessible
    • 2.1.1 Keyboard Navigability

      All functionality of the content is operable through a keyboard interface without requiring specific timings for individual keystrokes, except where the underlying function requires input that depends on the path of the user's movement and not just the endpoints.

      • Actions: Navigation menus and event links can be reached and activated through keyboard inputs.
  • 2.4 Navigable
    • 2.4.7 Focus Visible

      Any keyboard operable user interface has a mode of operation where the keyboard focus indicator is visible.

      • Actions: Buttons, forms, and links include clear focus outlines, helping users with motor impairments know where they are on the page.

Principle: Understandable

  • 3.1 Readable
    • 3.1.5 Reading Level

      When text requires reading ability more advanced than the lower secondary education level after removal of proper names and titles, supplemental content, or a version that does not require reading ability more advanced than the lower secondary education level, is available.

      • Actions: Content avoids jargon and uses simple, inclusive language so diverse community members, including those with different literacy levels, can understand resources and stories.
  • 3.2 Predictable
    • 3.2.3 Consistent Navigation

      Navigational mechanisms that are repeated on multiple web pages within a set of web pages occur in the same relative order each time they are repeated, unless a change is initiated by the user.

      • Actions: Place navigation bar at the top of every page with the same order.
    • 3.2.4 Consistent Identification (Layout & UI Behaviour)

      Components that have the same functionality within a set of web pages are identified consistently.

      • Actions: Use consistent naming across pages for similar features (e.g., Discussion Board, Profile).
  • 3.3 Input Assistance
    • 3.3.1 Error Identification (& Suggestions)

      If an input error is automatically detected, the item that is in error is identified and the error is described to the user in text.

      • Actions: Forms highlight specific errors and explain how to fix them.
    • 3.3.2 Labels (& Instructions)

      Labels or instructions are provided when content requires user input.

      • Actions: Add visible labels for all form fields; Use aria labels or aria-describedby for screen reader.
    • 3.3.4 Error Prevention and Recovery

      For web pages that cause legal commitments or financial transactions for the user to occur, that modify or delete user-controllable data in data storage systems, or that submit user test responses.

      • Actions: Confirmation steps and clear error prevention messages reduce mistakes and ensure users feel confident contributing to the site.
